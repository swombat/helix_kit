# Search Messages Across Chats

**Date**: 2026-03-01
**Status**: Ready for implementation

## Executive Summary

Add the ability to search through all message content within an account's chats. A search box in the navbar submits a query that renders a dedicated search results page. Each result shows contextual snippets (the line before and after the match), the chat title, and a timestamp. Clicking a result navigates to the parent chat.

The implementation follows the existing architecture closely: a collection action on `ChatsController`, a model scope on `Message`, a new Svelte page component, and a small addition to the navbar. No new gems, no new npm packages, no migrations.

## Architecture Overview

```
User types query in navbar search box
  -> GET /accounts/:account_id/chats/search?q=term
  -> ChatsController#search
  -> Message.search_in_account(account, query)
  -> Render Inertia "chats/search" with results + pagination
  -> Svelte page displays results with snippets
  -> Click result -> router.visit(accountChatPath(accountId, chatId))
```

### 2. Model Scope

- [ ] Add `Message.search_in_account` class method

**File**: `/Users/danieltenner/dev/helix_kit/app/models/message.rb`

Add a class method (not a scope, since it takes multiple arguments and returns a decorated result set). Place it after the existing scopes, before the callbacks:

```ruby
def self.search_in_account(account, query)
  return none if query.blank?

  joins(:chat)
    .where(chats: { account_id: account.id, discarded_at: nil })
    .where("messages.content ILIKE ?", "%#{sanitize_sql_like(query)}%")
    .where(role: %w[user assistant])
    .includes(:chat)
    .order(created_at: :desc)
end
```

Key decisions:
- Uses `ILIKE` for case-insensitive matching (PostgreSQL native, no gem needed).
- Excludes discarded chats via `discarded_at: nil` (matches `kept` scope behavior).
- Includes both archived and active chats (searching archived content is useful).
- Only searches `user` and `assistant` role messages (excludes `system` and `tool` noise).
- Orders by newest first so recent conversations surface at the top.
- Uses `sanitize_sql_like` to escape `%`, `_`, and `\` in the user's query.
- Uses `includes(:chat)` to avoid N+1 when rendering results.

Also add a helper instance method on `Message` to extract context lines around a match:

```ruby
def snippet_around(query)
  return content.truncate(200) if content.blank? || query.blank?

  lines = content.lines
  match_index = lines.index { |line| line.downcase.include?(query.downcase) }

  return content.truncate(200) unless match_index

  start = [match_index - 1, 0].max
  finish = [match_index + 1, lines.length - 1].min

  lines[start..finish].map(&:strip).join("\n").truncate(300)
end
```

### 3. Controller Action

- [ ] Add `search` action to `ChatsController`

**File**: `/Users/danieltenner/dev/helix_kit/app/controllers/chats_controller.rb`

Add the `search` action to the controller and add `:search` to the `except` list of the `set_chat` before_action (since search doesn't operate on a single chat):

```ruby
before_action :set_chat, except: [ :index, :create, :new, :search ]
```

Then the action:

```ruby
def search
  query = params[:q].to_s.strip
  results = []

  if query.present?
    scope = Message.search_in_account(current_account, query)
    @pagy, messages = pagy(scope, limit: 20)
    results = messages.map { |message| search_result_json(message, query) }
  end

  render inertia: "chats/search", props: {
    query: query,
    results: results,
    pagination: search_pagination,
    account: current_account.as_json
  }
end
```

Add the private helper methods:

```ruby
def search_result_json(message, query)
  {
    id: message.to_param,
    chat_id: message.chat.to_param,
    chat_title: message.chat.title_or_default,
    snippet: message.snippet_around(query),
    author_name: message.author_name,
    role: message.role,
    created_at: message.created_at.strftime("%b %d, %Y at %l:%M %p")
  }
end

def search_pagination
  return {} unless @pagy

  {
    count: @pagy.count,
    page: @pagy.page,
    pages: @pagy.pages,
    last: @pagy.last,
    from: @pagy.from,
    to: @pagy.to,
    prev: @pagy.prev,
    next: @pagy.next,
    series: @pagy.series.collect(&:to_s),
    per_page: @pagy.vars[:limit].to_s
  }
end
```

This mirrors the existing pagination pattern from `Admin::AuditLogsController`.

### 4. Search Results Page

- [ ] Create `app/frontend/pages/chats/search.svelte`

**File**: `/Users/danieltenner/dev/helix_kit/app/frontend/pages/chats/search.svelte`

This is the main page component. It receives `query`, `results`, `pagination`, and `account` as props from the controller.

```svelte
<script>
  import { router } from '@inertiajs/svelte';
  import { MagnifyingGlass, ChatText, ArrowRight } from 'phosphor-svelte';
  import PaginationNav from '$lib/components/navigation/PaginationNav.svelte';
  import { accountChatPath, searchAccountChatsPath } from '@/routes';

  let { query = '', results = [], pagination = {}, account } = $props();

  let searchInput = $state(query);
  let currentPage = $state(pagination.page || 1);

  function handleSearch(event) {
    event?.preventDefault();
    if (!searchInput.trim()) return;

    router.get(searchAccountChatsPath(account.id), { q: searchInput.trim() }, {
      preserveState: false,
    });
  }

  function handleKeydown(event) {
    if (event.key === 'Enter') {
      handleSearch(event);
    }
  }

  function goToChat(chatId) {
    router.visit(accountChatPath(account.id, chatId));
  }

  function handlePageChange(newPage) {
    router.get(
      searchAccountChatsPath(account.id),
      { q: query, page: newPage },
      { preserveState: false }
    );
  }

  function highlightMatch(text, term) {
    if (!term || !text) return text;
    const escaped = term.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
    const regex = new RegExp(`(${escaped})`, 'gi');
    return text.replace(regex, '<mark class="bg-yellow-200 dark:bg-yellow-800 rounded px-0.5">$1</mark>');
  }
</script>

<svelte:head>
  <title>{query ? `Search: ${query}` : 'Search'}</title>
</svelte:head>

<div class="max-w-3xl mx-auto px-4 md:px-6 py-8">
  <h1 class="text-2xl font-semibold mb-6">Search Messages</h1>

  <form onsubmit={handleSearch} class="mb-8">
    <div class="relative">
      <MagnifyingGlass
        size={20}
        class="absolute left-3 top-1/2 -translate-y-1/2 text-muted-foreground" />
      <input
        type="text"
        bind:value={searchInput}
        onkeydown={handleKeydown}
        placeholder="Search across all messages..."
        class="w-full pl-10 pr-4 py-2.5 border border-input rounded-lg bg-background
               text-sm focus:outline-none focus:ring-2 focus:ring-ring focus:border-transparent" />
    </div>
  </form>

  {#if query}
    {#if results.length === 0}
      <div class="text-center py-12 text-muted-foreground">
        <MagnifyingGlass size={48} class="mx-auto mb-4 opacity-30" />
        <p class="text-lg">No results found for "{query}"</p>
        <p class="text-sm mt-1">Try a different search term</p>
      </div>
    {:else}
      <p class="text-sm text-muted-foreground mb-4">
        {pagination.count} {pagination.count === 1 ? 'result' : 'results'} for "{query}"
      </p>

      <div class="space-y-3">
        {#each results as result (result.id)}
          <button
            onclick={() => goToChat(result.chat_id)}
            class="w-full text-left p-4 border border-border rounded-lg bg-card
                   hover:bg-muted/50 transition-colors group cursor-pointer">
            <div class="flex items-center justify-between mb-2">
              <div class="flex items-center gap-2 text-sm font-medium text-foreground">
                <ChatText size={16} class="text-muted-foreground" />
                <span class="truncate">{result.chat_title}</span>
              </div>
              <ArrowRight
                size={14}
                class="text-muted-foreground opacity-0 group-hover:opacity-100 transition-opacity" />
            </div>
            <div
              class="text-sm text-muted-foreground whitespace-pre-line leading-relaxed line-clamp-4">
              {@html highlightMatch(result.snippet, query)}
            </div>
            <div class="mt-2 text-xs text-muted-foreground/60">
              {result.author_name} &middot; {result.created_at}
            </div>
          </button>
        {/each}
      </div>

      <PaginationNav
        {pagination}
        bind:currentPage
        onPageChange={handlePageChange}
        class="mt-6" />
    {/if}
  {:else}
    <div class="text-center py-12 text-muted-foreground">
      <MagnifyingGlass size={48} class="mx-auto mb-4 opacity-30" />
      <p>Enter a search term to find messages across all your conversations</p>
    </div>
  {/if}
</div>
```

Key design decisions:
- Reuses the existing `PaginationNav` component (same pattern as admin audit logs).
- Uses `highlightMatch` with `{@html}` to render highlighted query terms. The query is regex-escaped to prevent injection. The HTML output only inserts `<mark>` tags around matches.
- Uses `line-clamp-4` to limit snippet display height.
- Clicking a result navigates to the chat via `accountChatPath`.
- The search form is a simple text input (no dropdown, no overlay) matching the "dedicated search results page" requirement.

## Design Decisions

**Why ILIKE and not full-text search?**
The requirement says "just straight search through all the messages." ILIKE is the simplest approach that handles case-insensitive matching. Full-text search (tsvector/tsquery) would add complexity (migrations for GIN indices, stemming configuration) that is not warranted for this initial implementation. If search performance becomes an issue with large datasets, a GIN index on `messages.content` or a move to `pg_trgm` can be added later without changing the API.

**Why Pagy for pagination?**
The application already uses Pagy (included in `ApplicationController`). The audit logs page establishes the pattern for pagination props and the `PaginationNav` Svelte component. Reusing this pattern keeps the codebase consistent.

**Why a collection action on ChatsController instead of a separate SearchController?**
Search is conceptually scoped to chats within an account. The route `/accounts/:id/chats/search` reads naturally and follows Rails resource nesting conventions. A separate controller would be over-engineering for a single action.

**Why not search chat titles or summaries?**
The requirements explicitly state: "Search scope: Message content only (not chat titles or summaries)." The chat title is displayed in each result for context, but the match is always against message content.

**Why 20 results per page?**
Each result includes a multi-line snippet, so fewer results per page is appropriate to avoid excessive scrolling. 20 strikes a balance between information density and usability.

## Edge Cases and Error Handling

- **Empty query**: Renders the search page with the search input and a prompt to enter a term. No database query is executed.
- **No results**: Shows a friendly "no results" empty state with the query echoed back.
- **SQL injection via search term**: `sanitize_sql_like` escapes LIKE wildcards. ActiveRecord parameterized queries handle the rest.
- **Very long search terms**: Truncated naturally by ILIKE performance (no explicit limit needed, but could add `params[:q].to_s.first(200)` if desired).
- **Discarded chats**: Excluded from results. Archived chats are included (they contain valuable searchable content).
- **Messages with nil content**: The `where(role: %w[user assistant])` filter combined with ILIKE matching naturally excludes empty-content messages.
- **HTML in search term highlight**: The `highlightMatch` function escapes regex special characters. The snippet itself comes from plain text message content (not pre-rendered HTML), so XSS risk is minimal. However, the snippet goes through `{@html}`, so the `<mark>` tag insertion is the only HTML added.
- **Account scoping**: The search is always scoped to `current_account` through the existing `AccountScoping` concern, preventing cross-account data leakage.

## Performance Considerations

For the initial implementation, `ILIKE` is sufficient. The query joins `messages` to `chats` and filters by `account_id` (indexed) and `discarded_at`. The main cost is the `ILIKE` scan on `content`.

If performance becomes a concern, these optimizations can be layered in later:
1. Add a GIN trigram index: `CREATE INDEX idx_messages_content_trgm ON messages USING gin (content gin_trgm_ops);`
2. Move to PostgreSQL full-text search with `tsvector` columns
3. Add Elasticsearch/Meilisearch for very large datasets

None of these would require changes to the controller or frontend API.
