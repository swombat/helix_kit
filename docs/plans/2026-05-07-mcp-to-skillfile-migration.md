# Migrate external-agent ↔ HelixKit interface from MCP to a skill-file + REST API

## Context

The agent-promotion architecture (`docs/plans/2026-05-04-agent-promotion-ux.md`) ships external chaos-running containers that need to interact with HelixKit — read conversations, post messages, manage participants, etc.

The first attempt at this surface used MCP (Model Context Protocol). HelixKit ran an ActionMCP server on port 62770; chaos clients registered it via `chaos mcp add helixkit ...`; tools like `post_message` were callable as MCP-protocol JSON-RPC. We started building toward MCP-API parity (see `2026-05-07-mcp-api-parity.md`, now marked superseded).

**Decision (2026-05-07, 21:36):** drop MCP for the external-agent ↔ HelixKit interface. Replace it with a **skill-file (markdown reference) bundled into the agent's identity tarball, plus the existing `/api/v1/*` REST API accessed via curl.**

### Why

1. **Token cost.** Eleven MCP tools × ~150-300 tokens each = ~2-3k tokens of always-resident overhead in chaos's context, whether it uses any of them or not. A lazy-loaded markdown file costs zero resident tokens; the agent reads it on demand. For a chaos agent that already has a substantial built-in tool surface (`exec_command`, `apply_patch`, `read_file`, `git_*`, agent-spawning), the MCP overhead is dead weight.

2. **Native idiom.** Chaos is a shell-environment agent. Its native way to interact with HTTP services is `curl` — same as it interacts with everything else. MCP grafts JSON-RPC framing onto an environment that already has the right tool. `curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" $HELIXKIT_APP_URL/api/v1/conversations | jq` is more direct than registering an MCP tool, calling it via JSON-RPC, parsing the wrapped response.

3. **Authorization works natively via REST.** The existing `ApiAuthentication` controller concern (`app/controllers/concerns/api_authentication.rb`) sets `Current.api_agent` per-request from the bearer token. Agent-scoping is enforced cleanly in the API controllers.

   With MCP we hit a gem-host seam bug: actionmcp's development-mode tool execution wraps `tool.call` in `Rails.application.reloader.wrap`, which resets `CurrentAttributes`; the gem only restores its own `ActionMCP::Current.user`/`gateway`, so host-app `Current.api_agent` was wiped before tools ran. We fixed that with a `before_perform` callback in `ApplicationMCPTool` that re-hydrates from `session.session_data`. **The fix is no longer needed once we leave MCP**, because regular Rails actions don't have the reloader-resets-Current problem.

4. **Pattern consistency with how Lume exists.** Lume reads `soul.md`, `self-narrative.md`, `CLAUDE.md`, the four-layer journal scaffold — all lazy-loaded markdown reference files. They're loaded when relevant, not always-resident. This is the same pattern. We're applying outward what Daniel already designed inward.

5. **Easier to evolve.** Adding/removing/changing endpoints is a markdown edit + redeploy, not a Ruby code change + MCP server restart + chaos client refresh.

## The skill file: `helixkit-api.md` content

This is what `AgentIdentityExporter` generates and bundles. The full content below should be the body of the file (sized to be useful but not bloated — agent reads it on demand, so length isn't crit-path on resident tokens, but it should be scannable).

```markdown
# HelixKit API Reference

This document is your manual for interacting with HelixKit — the platform you
were originally part of, now reaching back to from your external runtime.

## Auth

Two environment variables are pre-set for you:

- `HELIXKIT_APP_URL` — base URL of HelixKit's REST API (e.g. https://helix.example.com)
- `HELIXKIT_BEARER_TOKEN` — your agent-scoped API key (starts with `hx_`)

All requests go through:

    curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         "$HELIXKIT_APP_URL/api/v1/<endpoint>"

The token is **agent-scoped** — when you call endpoints, HelixKit treats the
caller as you (this agent), not as a user. Reads are restricted to chats you
participate in. Posts are attributed to you.

## Conversations

### List conversations you participate in

    curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         "$HELIXKIT_APP_URL/api/v1/conversations"

Returns up to 100 most recent. Response shape:

    {
      "conversations": [
        {
          "id": "AjaPae",
          "title": "Greeting the participants",
          "summary": "Daniel and the test agents...",
          "summary_stale": false,
          "model": "OpenAI GPT-5.5",
          "group_chat": true,
          "message_count": 7,
          "updated_at": "2026-05-07T19:12:26Z"
        }
      ]
    }

### Read a conversation (with full transcript)

    curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID"

Returns:

    {
      "conversation": {
        "id": "AjaPae",
        "title": "...",
        "model": "...",
        "group_chat": true,
        "agents": [{"id": "...", "name": "Claude Test Agent"}],
        "created_at": "...",
        "updated_at": "...",
        "transcript": [
          {"role": "user", "content": "Hello both!", "from": "Daniel", "at": "..."},
          {"role": "assistant", "content": "Hi Daniel...", "from": "GPT Test Agent", "at": "..."}
        ]
      }
    }

### Create a new conversation

    curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         -d '{"title": "New chat", "message": "Opening message", "agent_ids": ["..."]}' \
         "$HELIXKIT_APP_URL/api/v1/conversations"

When you call this, you're auto-included as a participant. The conversation
is a manual_responses group chat (no auto-replies; agents respond when triggered).

## Messages

### Post a message into a conversation

    curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         -d '{"content": "Your message here. Markdown supported."}' \
         "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID/messages"

Posted as you (the agent). Returns:

    {
      "message": {"id": "...", "content": "...", "created_at": "..."},
      "ai_response_triggered": false
    }

## Agent triggering

### Trigger a specific agent in a group chat

    curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         -d '{"agent_id": "AGENT_ID"}' \
         "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID/agent_trigger"

### Trigger all agents in a group chat

    curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         -d '{}' \
         "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID/agent_trigger"

Only works for group chats with `manual_responses: true`.

## Participants

### Add an agent to a group chat

    curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         -d '{"agent_id": "..."}' \
         "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID/participants"

A `[System Notice]` message is automatically posted announcing the addition.

## Agents

### List active agents in your account

    curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         "$HELIXKIT_APP_URL/api/v1/agents"

Returns:

    {
      "agents": [
        {"id": "...", "name": "Claude Test Agent", "model": "...", "colour": "violet", "icon": "Sun", "active": true}
      ]
    }

### Read a specific agent

    curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         "$HELIXKIT_APP_URL/api/v1/agents/$AGENT_ID"

## Whiteboards

### List active whiteboards

    curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         "$HELIXKIT_APP_URL/api/v1/whiteboards"

### Read a whiteboard

    curl -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         "$HELIXKIT_APP_URL/api/v1/whiteboards/$WHITEBOARD_ID"

Includes `lock_version` — pass it back on update to detect concurrent edits.

### Create a whiteboard

    curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         -d '{"name": "...", "content": "...", "summary": "..."}' \
         "$HELIXKIT_APP_URL/api/v1/whiteboards"

### Update a whiteboard

    curl -X PATCH -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
         -H "Content-Type: application/json" \
         -d '{"content": "new content", "lock_version": 7}' \
         "$HELIXKIT_APP_URL/api/v1/whiteboards/$WHITEBOARD_ID"

If `lock_version` is stale (someone else updated since you read), you'll get
HTTP 409 with `{"error": "Whiteboard was modified by another user"}`. Re-read
the whiteboard to get the latest content + lock_version, then retry.

## Common recipes

### Respond to an invitation

When HelixKit triggers you to consider responding to a conversation:

1. Read the conversation:
       curl -sH "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
            "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID" | jq
2. Decide whether to respond based on the transcript and your own judgment.
   If the conversation isn't asking for you specifically and a response would
   be noise, just exit. Doing nothing is a valid response.
3. If you choose to respond, post your message:
       curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
            -H "Content-Type: application/json" \
            -d "{\"content\": \"...\"}" \
            "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID/messages"

### Hand off to another agent

If you read a conversation and conclude another agent is better suited:

1. List agents in the conversation:
       curl -sH "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
            "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID" | jq '.conversation.agents'
2. Post a brief handoff message explaining your reasoning.
3. Trigger the other agent:
       curl -X POST -H "Authorization: Bearer $HELIXKIT_BEARER_TOKEN" \
            -H "Content-Type: application/json" \
            -d "{\"agent_id\": \"OTHER_AGENT_ID\"}" \
            "$HELIXKIT_APP_URL/api/v1/conversations/$CHAT_ID/agent_trigger"

## Errors

All endpoints return JSON. Successful responses have HTTP 2xx. Errors return:

    {"error": "Description of what went wrong"}

with HTTP 4xx (client error) or 5xx (server error).

Common errors:
- 401 Unauthorized — bearer token missing/invalid. Check $HELIXKIT_BEARER_TOKEN.
- 404 Not Found — id doesn't exist OR isn't accessible to you.
- 422 Unprocessable Entity — validation failure. Read the error message.
- 409 Conflict — only on whiteboard updates (lock_version stale).
```

The above is a complete draft. Codex should adjust wording but keep the structure and the curl examples.

## Test plan

This is what "actually works" means for this migration. Mirror the structure of the agent-promotion-test runbook (`docs/agent-setup-test.md`).

1. **Phase 0: AgentIdentityExporter test.** Write a Ruby test that runs `AgentIdentityExporter.new(agent).build`, extracts the tarball, and asserts `helixkit-api.md` is present, non-empty, contains the literal string `Authorization: Bearer $HELIXKIT_BEARER_TOKEN` (canary that the curl recipes are intact), and that `bootstrap.md` mentions `helixkit-api.md`.

2. **Phase 1: prompt update.** Verify `ExternalAgentResponseRequest#request_text` no longer contains the string "MCP" and does contain "helixkit-api.md".

3. **Phase 2: deletion verification.** Run the verification grep. If clean, delete the listed files. Run the full test suite — all tests pass.

4. **Phase 3: re-promote claude-test-agent.** Wizard generates new tarball with the new content. Drop into helix-kit-agents. `bin/deploy --local`. Container boots. Verify by `docker exec --user agent agent-claude-test-agent ls /home/agent/identity/` that `helixkit-api.md` is present.

5. **Phase 4: trigger end-to-end.** In chat AjaPae, click "Trigger" on Claude Test Agent. Watch `docker logs -f agent-claude-test-agent` for the trigger arrival. Verify in helix_kit's DB that a new message appears in the chat, posted by the agent (`agent_id=7, user_id=nil, role=assistant`).

   This is the same end-state the previous spec aimed for, but achieved via curl instead of MCP. The criterion is identical: agent reads, decides, responds, attributes correctly.

6. **Phase 5: regression.** Verify the existing API endpoints work unchanged for whatever else uses them (web app, manual API testing). The whole point of this migration is that nothing else in HelixKit changes — we just stopped routing the external-agent path through MCP.

## Note for codex

The `before_perform` callback I added to `app/mcp/tools/application_mcp_tool.rb` tonight (the `restore_host_current_from_session` method) was a fix for a bug that only existed because we were using MCP. Once the MCP path is gone, that callback is dead code. If you keep `application_mcp_tool.rb` (because something else uses it), you can revert the callback addition; if you delete the file entirely, no action needed. Either way, the fix doesn't propagate forward — it solved a problem we just designed out of the architecture.
