# Search Messages Across Chats

**Date**: 2026-03-01
**Status**: Ready for implementation
**Revision**: B (incorporates DHH feedback on revision A)

## Executive Summary

Add the ability to search through all message content within an account's chats. A search box in the navbar submits a query that renders a dedicated search results page. Each result shows contextual snippets (the line before and after the match), the chat title, and a timestamp. Clicking a result navigates to the parent chat.

The implementation follows the existing architecture closely: a collection action on `ChatsController`, a model scope on `Message`, a new Svelte page component, and a small addition to the navbar. No new gems, no new npm packages, no migrations.

### Changes from Revision A

Per DHH review feedback:
1. **XSS fix**: HTML entities are escaped in snippets before inserting `<mark>` tags
2. **Extracted shared pagination helper**: `pagy_to_hash` added to `ApplicationController`, replacing duplicate code in `Admin::AuditLogsController`
3. **`snippet_around` moved to controller**: it is a presentation concern, not a model concern

5. **Removed redundant `handleKeydown`**: the form `onsubmit` already handles Enter
6. **Removed unused `currentPage` state**: `PaginationNav` manages its own display
7. **Fixed time format**: `%-l` instead of `%l` to avoid leading space
8. **Simplified controller action**: fewer local variables, `(@messages || []).map` pattern
9. **Removed redundant `account` prop**: `account` is already in shared Inertia data via `ApplicationController`

## Architecture Overview

```
User types query in navbar search box
  -> GET /accounts/:account_id/chats/search?q=term
  -> ChatsController#search
  -> Message.search_in_account(account, query)
  -> Render Inertia "chats/search" with results + pagination
  -> Svelte page displays results with snippets
  -> Click result -> router.visit(accountChatPath(accountId, chatId))
```

### 2. Model Scope

- [ ] Add `Message.search_in_account` class method

**File**: `/Users/danieltenner/dev/helix_kit/app/models/message.rb`

Add a class method after the existing scopes, before the callbacks:

```ruby
def self.search_in_account(account, query)
  return none if query.blank?

  joins(:chat)
    .where(chats: { account_id: account.id, discarded_at: nil })
    .where("messages.content ILIKE ?", "%#{sanitize_sql_like(query)}%")
    .where(role: %w[user assistant])
    .includes(:chat)
    .order(created_at: :desc)
end
```

Key decisions:
- Uses `ILIKE` for case-insensitive matching (PostgreSQL native, no gem needed).
- Excludes discarded chats via `discarded_at: nil` (matches `kept` scope behavior).
- Includes both archived and active chats (searching archived content is useful).
- Only searches `user` and `assistant` role messages (excludes `system` and `tool` noise).
- Orders by newest first so recent conversations surface at the top.
- Uses `sanitize_sql_like` to escape `%`, `_`, and `\` in the user's query.
- Uses `includes(:chat)` to avoid N+1 when rendering results.

No `snippet_around` method on the model -- snippet extraction is a presentation concern handled in the controller (see step 3).

```ruby
def search
  @query = params[:q].to_s.strip

  if @query.present?
    @pagy, @messages = pagy(Message.search_in_account(current_account, @query), limit: 20)
  end

  render inertia: "chats/search", props: {
```

Add the private helper methods:

```ruby
def search_result_json(message, query)
  {
    id: message.to_param,
    chat_id: message.chat.to_param,
    chat_title: message.chat.title_or_default,
    snippet: snippet_around(message.content, query),
    author_name: message.author_name,
    role: message.role,
    created_at: message.created_at.strftime("%b %-d, %Y at %-l:%M %p")
  }
end

def snippet_around(content, query)
  return content.truncate(200) if content.blank? || query.blank?

  lines = content.lines
  match_index = lines.index { |line| line.downcase.include?(query.downcase) }
  return content.truncate(200) unless match_index

  start = [match_index - 1, 0].max
  finish = [match_index + 1, lines.length - 1].min
  lines[start..finish].map(&:strip).join("\n").truncate(300)
end
```

Key changes from revision A:
- `snippet_around` is a private controller method, not a model method. It takes `content` and `query` as arguments -- pure data in, string out. This is where presentation logic belongs.
- Time format uses `%-d` and `%-l` to strip leading spaces (produces "Mar 1, 2026 at 3:45 PM" not "Mar 01, 2026 at  3:45 PM").
- The controller action uses instance variables and `(@messages || []).map` to avoid the `results = []` / conditional-populate ceremony.
- No `account` prop -- it is already shared.

### 5. Search Results Page

```svelte
<script>
  import { page, router } from '@inertiajs/svelte';
  import { MagnifyingGlass, ChatText, ArrowRight } from 'phosphor-svelte';
  import PaginationNav from '$lib/components/navigation/PaginationNav.svelte';
  import { accountChatPath, searchAccountChatsPath } from '@/routes';

  let { query = '', results = [], pagination = {} } = $props();

  const account = $derived($page.props.account);

  let searchInput = $state(query);

  function handleSearch(event) {
    event?.preventDefault();
    if (!searchInput.trim()) return;

    router.get(searchAccountChatsPath(account.id), { q: searchInput.trim() }, {
      preserveState: false,
    });
  }

  function goToChat(chatId) {
    router.visit(accountChatPath(account.id, chatId));
  }

  function handlePageChange(newPage) {
    router.get(
      searchAccountChatsPath(account.id),
      { q: query, page: newPage },
      { preserveState: false }
    );
  }

  function escapeHtml(text) {
    const div = document.createElement('div');
    div.textContent = text;
    return div.innerHTML;
  }

  function highlightMatch(text, term) {
    if (!term || !text) return escapeHtml(text || '');
    const escaped = escapeHtml(text);
    const termEscaped = term.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
    const regex = new RegExp(`(${termEscaped})`, 'gi');
    return escaped.replace(regex, '<mark class="bg-yellow-200 dark:bg-yellow-800 rounded px-0.5">$1</mark>');
  }
</script>
```

## Design Decisions

**Why ILIKE and not full-text search?**
The requirement says "just straight search through all the messages." ILIKE is the simplest approach that handles case-insensitive matching. Full-text search (tsvector/tsquery) would add complexity (migrations for GIN indices, stemming configuration) that is not warranted for this initial implementation. If performance becomes a concern, optimizations can be layered in later without changing the controller or frontend API.

**Why Pagy for pagination?**
The application already uses Pagy (included in `ApplicationController`). The audit logs page establishes the pattern for pagination props and the `PaginationNav` Svelte component. The new shared `pagy_to_hash` helper eliminates the duplication.

**Why a collection action on ChatsController instead of a separate SearchController?**
Search is conceptually scoped to chats within an account. The route `/accounts/:id/chats/search` reads naturally and follows Rails resource nesting conventions. A separate controller would be over-engineering for a single action.

**Why `snippet_around` in the controller, not the model?**
The snippet takes a search query -- something the Message model should know nothing about -- and produces a display-oriented string. This is presentation logic. The Message model is already 600+ lines. Keeping it in the controller as a private method is appropriate for something this small and single-purpose.

**Why no `account` prop?**
`ApplicationController` already shares `account: current_account&.as_json` with every authenticated Inertia page via `inertia_share`. The Svelte component accesses it as `$page.props.account`. Passing it again as a page-specific prop would be redundant.

**Why 20 results per page?**
Each result includes a multi-line snippet, so fewer results per page is appropriate to avoid excessive scrolling. 20 strikes a balance between information density and usability.

## Edge Cases and Error Handling

- **Empty query**: Renders the search page with the search input and a prompt to enter a term. No database query is executed.
- **No results**: Shows a friendly "no results" empty state with the query echoed back.
- **SQL injection via search term**: `sanitize_sql_like` escapes LIKE wildcards. ActiveRecord parameterized queries handle the rest.
- **Very long search terms**: Truncated naturally by ILIKE performance (no explicit limit needed, but could add `params[:q].to_s.first(200)` if desired).
- **Discarded chats**: Excluded from results. Archived chats are included (they contain valuable searchable content).
- **Messages with nil content**: The `where(role: %w[user assistant])` filter combined with ILIKE matching naturally excludes empty-content messages.
- **XSS via `{@html}`**: The `highlightMatch` function calls `escapeHtml` on the snippet text first, converting `<`, `>`, `&`, and `"` to HTML entities. Only then does it insert `<mark>` tags around matched terms. User content cannot inject arbitrary HTML.
- **Account scoping**: The search is always scoped to `current_account` through the existing `AccountScoping` concern, preventing cross-account data leakage.

## Performance Considerations

For the initial implementation, `ILIKE` is sufficient. The query joins `messages` to `chats` and filters by `account_id` (indexed) and `discarded_at`. The main cost is the `ILIKE` scan on `content`.

If performance becomes a concern, these optimizations can be layered in later:
1. Add a GIN trigram index: `CREATE INDEX idx_messages_content_trgm ON messages USING gin (content gin_trgm_ops);`
2. Move to PostgreSQL full-text search with `tsvector` columns
3. Add Elasticsearch/Meilisearch for very large datasets

None of these would require changes to the controller or frontend API.
